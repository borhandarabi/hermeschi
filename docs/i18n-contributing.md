# مشارکت در ترجمه‌های رابط کاربری

HermesChi از یک ماژول i18n سبک و بدون وابستگی استفاده می‌کند که از زبان‌های انگلیسی (`en`) و فارسی (`fa`) پشتیبانی می‌کند. فارسی راست‌به‌چپ و انگلیسی چپ‌به‌راست رندر می‌شود. جهت سند به locale فعال گره خورده است، بنابراین تعویض زبان در تنظیمات بلافاصله چیدمان را تغییر می‌دهد.

## فایل ترجمه

تمام وضعیت ترجمه در این فایل قرار دارد:

```text
src/lib/i18n.ts
```

بخش‌های مهم این فایل عبارت‌اند از:

- `LocaleId`: نوع union از شناسهٔ localeهای پشتیبانی‌شده (`'en' | 'fa'`).
- `EN`: کلیدهای انگلیسی به‌عنوان منبع حقیقت. هر کلید جدید اول اینجا اضافه می‌شود.
- `FA`: جدول ترجمهٔ فارسی، که هر کلید در `EN` را پوشش می‌دهد.
- `RTL_LOCALES`: یک `Set<LocaleId>` که مشخص می‌کند کدام localeها به‌صورت RTL رندر می‌شوند. در حال حاضر `{'fa'}`.
- `LOCALES`: نگاشت از شناسهٔ locale به جدول ترجمه.
- `LOCALE_LABELS`: برچسب‌های قابل‌خواندن توسط انسان که در انتخابگر زبان نمایش داده می‌شوند.
- `getDir(locale)`: برای فارسی `'rtl'` و برای بقیه `'ltr'` برمی‌گرداند.
- `applyDocumentDir(locale)`: ویژگی‌های `dir`، `lang` و `data-dir` را روی `<html>` می‌نویسد.
- `setLocale(id)`: locale را در `localStorage` ذخیره می‌کند، `applyDocumentDir` را صدا می‌زند و رویداد `'locale-change'` را ارسال می‌کند تا مشترک‌ها (subscriberها) دوباره رندر شوند.
- `t(key, params?)`: رشتهٔ ترجمه‌شده برای locale فعال را با placeholderهای `{placeholder}` اختیاری برمی‌گرداند.

## localeهای پشتیبانی‌شده

| شناسهٔ locale | برچسب    | جهت | وضعیت               |
| --------- | --------- | --------- | ------------------ |
| `en`      | English   | LTR       | منبع حقیقت          |
| `fa`      | فارسی     | RTL       | در حال ترجمه فعال |

برای افزودن یک locale جدید:

۱. شناسهٔ آن را به نوع union `LocaleId` اضافه کنید.
۲. اگر راست‌به‌چپ رندر می‌شود (عربی، عبری، اردو و غیره)، آن را به مجموعهٔ `RTL_LOCALES` اضافه کنید.
۳. یک جدول ترجمه بسازید که `LocaleTranslations` را ارضا کند (هر کلید در `EN` باید حضور داشته باشد).
۴. آن را در `LOCALES` و `LOCALE_LABELS` ثبت کنید.
۵. یک مورد آزمون به `src/lib/i18n.test.ts` اضافه کنید که ادعا کند حداقل چند کلید تحت locale جدید به‌درستی resolve می‌شوند.

## افزودن یا بهبود ترجمه‌های فارسی

۱. `src/lib/i18n.ts` را باز کنید.
۲. شیء `FA` را پیدا کنید.
۳. مقدار سمت راست هر کلید را به‌روزرسانی کنید. از فارسی استاندارد ایرانی و نیم‌فاصله‌های صحیح استفاده کنید (مثلاً `نمایه‌ها`، نه `نمایه ها`).

نمونه:

```ts
const FA: LocaleTranslations = {
  'nav.dashboard': 'داشبورد',
  'nav.chat': 'گفتگو',
}
```

نام کلیدها را دقیقاً یکسان نگه دارید. فقط متن ترجمه‌شده را ویرایش کنید.

## افزودن متن جدید رابط کاربری قابل‌ترجمه

اگر متن انگلیسیِ hardcoded در رابط کاربری دیدید:

۱. یک کلید جدید به `EN` (منبع حقیقت) اضافه کنید.
۲. همان کلید را به جدول `FA` اضافه کنید.
۳. متن hardcoded در کامپوننت را با `t('your.newKey')` جایگزین کنید.

نمونه:

```ts
// src/lib/i18n.ts
const EN = {
  'common.retry': 'Retry',
} as const

const FA: LocaleTranslations = {
  'common.retry': 'تلاش دوباره',
}
```

سپس در کامپوننت:

```tsx
import { t } from '@/lib/i18n'

;<button>{t('common.retry')}</button>
```

چون `LocaleTranslations = Record<TranslationKey, string>` و `TranslationKey = keyof typeof EN` است، TypeScript از کامپایل شدن جلوگیری می‌کند اگر جدول `FA` کلیدی را جا بیندازد. این رفتار عمدی است تا جدول فارسی از `EN` جدا نشود.

## جای‌گذاری

برای رشته‌هایی که شامل مقادیر پویا هستند، از توکن‌های `{placeholder}` در رشتهٔ ترجمه استفاده کنید و `params` را به‌عنوان آرگومان دوم به `t()` بدهید:

```ts
// src/lib/i18n.ts
const EN = {
  'chat.messageList.unreadCount': '{count} unread',
} as const

const FA: LocaleTranslations = {
  'chat.messageList.unreadCount': '{count} خوانده‌نشده',
}
```

```tsx
import { t } from '@/lib/i18n'

;<span>{t('chat.messageList.unreadCount', { count: 5 })}</span>
// EN: '5 unread'
// FA: '5 خوانده‌نشده'
```

چندین placeholder هم پشتیبانی می‌شوند:

```ts
t('chat.contextBar.tokens', { count: 500, max: 8000 })
// EN: '500 / 8000 tokens'
// FA: '500 / 8000 توکن'
```

placeholderهای بدون تطابق (یعنی کلید متناظر در `params` ندارند) دست‌نخورده باقی می‌مانند؛ بنابراین مترجم‌ها می‌توانند در صورت لزوم از `{braces}` تحت‌اللفظی در متن استفاده کنند.

اعداد با `String(value)` تبدیل به رشته می‌شوند، نه با `Intl.NumberFormat`. این تصمیم عمدی است تا فراخوان کنترل کامل روی نمایش ارقام داشته باشد. بیشتر رابط‌های فنی حتی در حالت فارسی ارقام ASCII را ترجیح می‌دهند تا در شناسه‌های copy-pance ترکیبی از ارقام لاتین و فارسی پیش نیاید.

## قراردادهای RTL

اپ به‌صورت ذاتی RTL-aware است. هنگام نوشتن کامپوننت‌ها از این قوانین پیروی کنید:

### از ویژگی‌های منطقی CSS استفاده کنید

Tailwind v4 با ابزارهای ویژگی منطقی عرضه می‌شود که به‌طور خودکار بر اساس ویژگی `dir` عمل می‌کنند. همیشه این‌ها را به‌جای همتایان فیزیکی ترجیح دهید:

| فیزیکی (پرهیز)   | منطقی (استفاده) | ویژگی CSS               |
| ---------------- | -------------- | ----------------------- |
| `ml-`            | `ms-`          | margin-inline-start     |
| `mr-`            | `me-`          | margin-inline-end       |
| `pl-`            | `ps-`          | padding-inline-start    |
| `pr-`            | `pe-`          | padding-inline-end      |
| `left-`          | `start-`       | inset-inline-start      |
| `right-`         | `end-`         | inset-inline-end        |
| `text-left`      | `text-start`   | text-align: start       |
| `text-right`     | `text-end`     | text-align: end         |
| `border-l`       | `border-s`     | border-inline-start     |
| `border-r`       | `border-e`     | border-inline-end       |
| `rounded-l-*`    | `rounded-s-*`  | border-start-radius     |
| `rounded-r-*`    | `rounded-e-*`  | border-end-radius       |

### از variantهای `rtl:` / `ltr:` برای overrideهای یک‌باره استفاده کنید

وقتی یک ویژگی منطقی کافی نیست (مثلاً یک انیمیشن که نیاز به برگرداندن جهت دارد، یا یک transform که فقط باید در یک جهت اعمال شود)، از variantهای داخلی `rtl:` و `ltr:` در Tailwind استفاده کنید:

```tsx
<div className="animate-in slide-in-from-right-5 rtl:slide-in-from-left-5">
```

اعلان‌های `@variant rtl` و `@variant ltr` در `src/styles.css` این نیت را صریحاً مستند می‌کنند.

### از `useLocaleDirection()` برای منطق واکنش‌گرِ آگاه از جهت استفاده کنید

برای کامپوننت‌هایی که در زمان اجرا نیاز دارند جهت فعال را بدانند (مثلاً انتخاب بین دو چیدمان متفاوت، یا محاسبهٔ یک موقعیت)، از hook واکنش‌گرا استفاده کنید:

```tsx
import { useLocaleDirection } from '@/hooks/use-locale-direction'

function MyComponent() {
  const { isRtl, dir } = useLocaleDirection()
  return <div style={{ flexDirection: isRtl ? 'row-reverse' : 'row' }}>...</div>
}
```

این hook روی رویداد `'locale-change'` مشترک می‌شود و وقتی کاربر زبان را تعویض می‌کند، به‌طور خودکار دوباره رندر می‌شود. همچنین روی رویدادهای `storage` بین‌زبانه‌ای هم‌گام می‌شود.

### `dir` را خودتان تنظیم نکنید

ویژگی `dir` روی `<html>` منبع حقیقت واحد است. این ویژگی در سه جا تنظیم می‌شود:

۱. توسط `themeScript`.inline در `src/routes/__root.tsx` قبل از hydration ری‌اکت (تا نخستین paint هم در جهت درست باشد).
۲. توسط `setLocale()` → `applyDocumentDir()` وقتی کاربر زبان را تعویض می‌کند.
۳. توسط یک `useEffect` در `RootLayout` که هر بار `useLocaleDirection()` تغییری را گزارش می‌کند، دوباره اعمالش می‌کند.

هرگز `<div dir="rtl">` یا `element.dir = 'rtl'` را در کد کامپوننت ننویسید. اگر یک ظرف آگاه از جهت نیاز دارید، به `dir` ارث‌رسیده از `<html>` تکیه کنید.

### از `Intl.DateTimeFormat('en-US', ...)` یا `toLocaleDateString('en-US', ...)` استفاده نکنید

سخت‌کد کردن `'en-US'` باعث می‌شود تاریخ‌ها فارغ از locale فعال، با فرمت انگلیسی آمریکایی رندر شوند. به‌جای آن یا:

- `undefined` به‌عنوان locale بفرستید تا مرورگر از `navigator.language` استفاده کند: `new Intl.DateTimeFormat(undefined, { ... })`
- یا locale فعال را صریحاً بفرستید: `new Intl.DateTimeFormat(getLocale(), { ... })`

رویکرد دوم زمانی بهتر است که بخواهید تاریخ از انتخاب locale کاربر در اپ پیروی کند (که ممکن است با locale مرورگر او متفاوت باشد).

### از کلاس‌های Tailwind ویژگی فیزیکی در کد جدید استفاده نکنید

یک قانون ESLint آینده روی کلاس‌های `ml-`، `mr-`، `pl-`، `pr-`، `left-`، `right-`، `text-left`، `text-right` در کد جدید هشدار خواهد داد. تا آن زمان، قبل از باز کردن PR، diff خود را برای این کلاس‌ها بازبینی کنید. می‌توانید تغییرات خود را با این دستور grep کنید:

```bash
git diff main...HEAD -- '*.tsx' '*.ts' | grep -E '^\+.*\b(ml-|mr-|pl-|pr-|left-|right-|text-left|text-right)\b'
```

اگر grep چیزی برگرداند، معادل ویژگی منطقی را ترجیح دهید، مگر دلیل خاصی داشته باشید (و آن را در کامنت کد مستند کنید).

## راهنمای اختصاصی فارسی

### ارقام

ارقام ASCII (`0123456789`) را نگه دارید در:

- بلاک‌های کد و کد inline
- شناسه‌ها، مسیرها، نام فایل‌ها
- شمارش توکن و سایر رشته‌های فنی رابط کاربری که کاربر ممکن است copy-paste کند
- پارامترهای جای‌گذاری (این‌ها با `String(value)` به رشته تبدیل می‌شوند)

ارقام فارسی (`۰۱۲۳۴۵۶۷۸۹`) را فقط در:

- متن نثر که عدد بخشی از جمله است (و فقط اگر تیم تصمیم بگیرد از ارقام فارسی استفاده کند — در حال حاضر برای یکدستی، ارقام ASCII را نگه می‌داریم)

### نشانه‌گذاری

- در متن فارسی از کامای فارسی `،` (U+060C) به‌جای کامای ASCII `,` استفاده کنید.
- به‌جای `؛` از `؛` (U+061B) به‌جای `;` برای نقطه‌ویرگول در متن فارسی استفاده کنید.
- به‌جای `""` از گیومهٔ فارسی «» برای نقل‌قول دور متن فارسی استفاده کنید.
- از `…` (U+2026) برای علامت حذف استفاده کنید — در هر دو حالت LTR و RTL درست رندر می‌شود.
- دو نقطه `:` در فارسی به همان شکل قابل‌استفاده است.

### نیم‌فاصله (ZWNJ)

فارسی از ZWNJ (`\u200c`، نامرئی رندر می‌شود) برای جداکردن واژگانی که نباید به هم بپیوندند استفاده می‌کند. همیشه آن را در جای درستش قرار دهید:

- ✅ `نمایه‌ها` (درست)
- ❌ `نمایه ها` (اشتباه — دو واژه به‌هم چسبیده به‌نظر می‌رسند)
- ✅ `می‌خواهم` (درست)
- ❌ `میخواهم` (اشتباه)

بیشتر IMEهای فارسی ZWNJ را به‌طور خودکار درج می‌کنند. اگر ترجمه‌ها را با چیدمان کیبورد غیرفارسی تایپ می‌کنید، مراقب باشید که آن‌ها را به‌طور دستی قرار دهید.

## آزمون محلی

اجرای:

```bash
pnpm exec vitest run src/lib/i18n.test.ts
pnpm exec tsc --noEmit
pnpm build
```

سپس به تنظیمات → زبان بروید و به فارسی تعویض کنید. راستی‌آزمایی کنید:

۱. عنصر `<html>` دارای `dir="rtl"` و `lang="fa"` است (با DevTools بررسی کنید).
۲. sidebar در سمت راست صفحه ظاهر می‌شود.
۳. تراز متن در صورت لزوم راست‌چین است.
۴. فونت Vazirmatn بارگذاری شده است (تب Fonts در DevTools را بررسی کنید).
۵. در بارگذاری اولیهٔ صفحه هیچ فلش LTR دیده نمی‌شود (`themeScript` باید `dir` را قبل از hydration تنظیم کرده باشد).

برای الگوهای آزمون RTL اختصاصی کامپوننت، به `src/lib/i18n.test.ts` نگاه کنید. آزمون‌هایی که `setLocale()` را اجرا می‌کنند در Node امن هستند (DOM لازم نیست) — assertionهای مبتنی بر DOM با `typeof document !== 'undefined'` محافظت می‌شوند.

## یادداشت‌های معماری

### چرا بدون ICU MessageFormat؟

ما به‌طور عمدی از `Intl.MessageFormat` یا هر نحو پیام به‌سبک ICU استفاده نمی‌کنیم. جمع‌بستن‌ها در محل فراخوانی با انتخاب یک کلید متفاوت مدیریت می‌شوند:

```ts
// EN
'messages.unread.one': '1 unread message',
'messages.unread.many': '{count} unread messages',

// محل فراخوانی
const key = count === 1 ? 'messages.unread.one' : 'messages.unread.many'
return t(key, { count })
```

این کار ماژول i18n را کوچک و بدون وابستگی نگه می‌دارد. بهای آن این است که مترجم‌ها باید چندین کلید برای فرم‌های جمع بسازند، ولی در عمل این مورد در رابط کاربری ما نادر است.

### چرا نه react-i18next؟

نیازهای i18n اپ ساده است: حدود ۲۵۰ کلید، دو locale، بدون lazy-loading، بدون تقسیم namespace. `react-i18next` حدود ۳۰KB به bundle اضافه می‌کند برای ویژگی‌هایی که استفاده نمی‌کنیم. تابع `t()` فعلی یک‌خطی است که از یک نقشهٔ درون‌حافظه‌ای از پیش ساخته‌شده می‌خواند؛ ساختن چیزی که به‌طور معناداری سریع‌تر باشد غیرممکن است.

اگر تعداد localeها از حدود ۵ فراتر رفت، یا به تقسیم namespace با lazy-loading نیاز داشتیم، این تصمیم را بازبینی کنید.

### چرا `themeScript` فهرست RTL_LOCALES را تکثیر می‌کند؟

`themeScript` در `src/routes/__root.tsx` به‌عنوان یک تگ `<script>` inline قبل از hydration ری‌اکت اجرا می‌شود. این اسکریپت به module graph دسترسی ندارد، بنابراین نمی‌تواند `RTL_LOCALES` را از `src/lib/i18n.ts` import کند. فهرست تکثیر شده (به‌همراه یک کامنت توضیحی) تا جهت سند قبل از نخستین paint تنظیم شود و از فلش LTR برای کاربران locale فارسی جلوگیری کند.

اگر یک locale RTL جدید اضافه کنید، باید هر دو را به‌روز کنید:
۱. `RTL_LOCALES` در `src/lib/i18n.ts`
۲. آرایهٔ `RTL_LOCALES` در رشتهٔ `themeScript` در `src/routes/__root.tsx`

مجموعه آزمون موجود drift میان این دو فهرست را شناسایی نمی‌کند. یک TODO پیگیری شده این است که بدنهٔ اسکریپت را در یک فایل `.ts` جداگانه استخراج کنیم که build آن را inline می‌کند، تا همان منبع در هر دو زمینه سرو شود.

## وضعیت فعلی

گروه‌های صفحهٔ زیر به `t()` مهاجرت کرده‌اند:

- ✅ صفحهٔ گفتگو (composer، header، حالت خالی، فهرست پیام، آیتم پیام، sidebar، دیالوگ ارائه‌دهنده‌ها، دیالوگ نشست‌ها، برچسب‌های ابزار)
- ✅ صفحهٔ تنظیمات (عناوین بخش‌ها، انتخابگر زبان)
- ✅ کامپوننت‌های مشترک (نوار تب موبایل، workspace shell)
- ✅ عناوین صفحهٔ Tasks/Profiles
- ✅ زیرساخت RTL (themeScript، useLocaleDirection، @variant rtl، فونت Vazirmatn)
- ✅ ماژول i18n (جای‌گذاری، getDir، applyDocumentDir)

گروه‌های صفحهٔ زیر هنوز رشته‌های انگلیسی hardcoded دارند (برای کار پیگیری برنامه‌ریزی شده‌اند):

- ⚠️ برچسب‌های ردیف و ورودی‌های فرم صفحهٔ تنظیمات (حدود ۲۵۰ رشته)
